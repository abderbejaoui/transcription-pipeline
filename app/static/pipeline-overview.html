Module Overview
What This Module Does

After the ASR model produces a raw transcript and the LLM DETECT stage flags suspicious spans, this module resolves each span to the correct canonical term. It operates entirely on text — no external LLM calls — making it fast, deterministic, and fully explainable.

Before any matching begins, CAMeL Tools normalises the Arabic text and a morphological analyser filters out real Arabic words so they are never sent for correction. The remaining spans pass through six correction stages in priority order, short-circuiting as soon as one stage reaches confidence threshold. The phonetic core — used across multiple stages — is a custom CEQ (Consonant Equivalence Class) + Editex + Jaro-Winkler scorer built specifically for Arabic→Latin drug-name transliteration.

CAMeL Tools — Arabic Normalisation & Morphological Filtering
correction.py · _normalize_arabic()  |  flag.py · _morph_is_real_arabic()
CAMeL Tools (Columbia Arabic Machine-learning Language toolkit) is used at two points before any correction stage runs:

(1) Text Normalisation — every input span is normalised using three CAMeL utilities so that dialectal and script variations do not prevent matches:
  • dediac_ar — removes all tashkeel (diacritics / harakat)
  • normalize_alef_ar — collapses أ / إ / آ / ا / ى → ا
  • normalize_teh_marbuta_ar — ة → ه (common in Gulf dialect)

(2) Morphological Filtering — the two CAMeL MorphologyDB databases (calima-glf-01 Gulf Arabic and calima-msa-r13 MSA) are queried for each word. Any word that returns at least one valid POS analysis is a genuine Arabic word and is excluded from correction entirely. Drug transliterations like لازيكس or بنادول produce zero analyses in both databases, so they correctly pass through to the correction cascade. This morphology gate runs at O(1) per word thanks to an LRU cache.

مريض → 3 valid analyses → skipped real Arabic word (patient)

Arabic Spelling Correction
arabic_spelling.py · weighted_arabic_spelling.py
Gulf Arabic ASR produces systematic, dialect-driven substitutions that are not random noise but predictable patterns. SymSpellPy is loaded with a custom frequency dictionary where these substitutions carry lower edit cost: ق (qaf) → ك (kaf), ذ (thal) → ز (zain), ث → س are all common in Gulf speech. The weighted costs ensure the corrector reaches the dialect-appropriate form first without over-correcting MSA terms that happen to share the same skeleton.

English Fuzzy Matching
correction.py — english_fuzzy_match()
For terms in Latin script the engine computes a composite score across all canonical terms and their aliases: 0.8 × rapidfuzz token_sort_ratio + 0.2 × jellyfish Jaro-Winkler. The token_sort_ratio handles transpositions and partial matches; Jaro-Winkler rewards shared prefix and penalises suffix differences — critical for brand names that share a stem (e.g. Metformin / Metformina). Matches above 0.75 are accepted; 0.60–0.75 are flagged for HITL review.
Examples

Arabic → English Skeleton Matching
arabic_matcher.py — HybridMatcher · skeleton()
Clinicians write English terms using Arabic phonetic approximation. The skeleton matcher handles this cross-script challenge in two steps:

(1) Extraction: after CAMeL normalisation, short vowels are dropped and each Arabic consonant is mapped to its closest Latin/IPA equivalent via a Buckwalter-derived table. Only the consonant sequence is retained.

(2) Matching: the same extraction is applied offline to every lexicon entry, building a skeleton index. At query time, the Arabic skeleton is compared against all English skeletons using normalised Levenshtein ratio. A threshold of 0.70 is used for acceptance.

Drug Name Normalization — CEQ · Editex · Jaro-Winkler
drug_normalize.py — _phonetic_similarity() · _editex() · _jaro_winkler()
Plain Levenshtein over consonant skeletons is orthographic: it treats a b↔p swap identically to a b↔z swap, even though the former is a trivial voiced/voiceless confusion and the latter is phonetically unrelated. This stage fixes that with a three-layer phonetic scoring system built specifically for Arabic→Latin drug names.

Layer 1 — CEQ (Consonant Equivalence Classes): every skeleton character is mapped to one of 14 phonetic class symbols. Within-class substitution costs 0; adjacent-class substitution costs 0.5; all other substitutions cost 1.0. The 14 classes and their neighbour pairs are derived from Buckwalter and IPA equivalences for Gulf Arabic.

a (ا أ إ ى ع)
Scoring Formula
score = 0.6 × (1 − editex(CEQ_a, CEQ_b) / max_len) + 0.4 × JW(CEQ_a, CEQ_b)
• Editex (Zobel & Dart 1996 adapted): dynamic programming over CEQ strings — within-class cost 0, adjacent-class cost 0.5, other cost 1.0
• Jaro-Winkler: standard Jaro score + prefix bonus p=0.1 (up to 4 matching prefix chars) — rewards brand names that share a common stem
• Weights: Editex carries 60% (phonetic substance); Jaro-Winkler carries 40% (prefix/order)

Multi-Word Arabic Phrase Matching
phonetic.py — match_multi_word_arabic() · combined_phonetic_similarity()
Single-term matchers fail when a multi-word English clinical expression is approximated as multiple Arabic tokens. This stage scans all n-gram windows (n=2 and n=3) of the input token list and scores each window with combined_phonetic_similarity: 0.5 × IPA Levenshtein (normalised) + 0.5 × consonant skeleton overlap. The Kneser-Ney n-gram LM provides an additional perplexity signal that down-ranks matches which are linguistically implausible in context.

Vector Lexicon Retrieval
vector_lexicon.py — VectorLexicon
The broadest and most permissive stage — a semantic safety net. All lexicon terms (canonical + aliases) are indexed at startup as character n-gram TF-IDF vectors (n=2–4), which capture sub-word phonetic similarity without a GPU. An optional transformer backend replaces this with multilingual sentence-transformer embeddings for true semantic matching. The input span is vectorised and the top-K nearest neighbours are retrieved via FAISS flat cosine index; only results above cosine similarity 0.60 are returned.
FAISS flat cosine index n-gram TF-IDF (n=2–4) sentence-transformers (optional) Min cosine similarity: 0.60

All correction stages share a single knowledge base: data/medical_lexicon.jsonl (~200 curated entries). Each entry defines a canonical term, Gulf Arabic aliases, a type tag, and a priority weight. Aliases cover the phonetic approximations most frequently produced by Gulf Arabic clinical dictation.
